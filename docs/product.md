# Hearth 产品文档

**产品定位**：面向家庭 Homelab 的本地运维中枢。  
**核心承诺**：让 1–2 位家庭管理者在几秒内掌控全局，并在异常时快速定位和进入处置。

## 1. 用户、问题与边界

### 目标用户

主要用户是维护 NAS、PVE、Docker 服务、路由器和局域网设备的家庭管理者。产品不以多人协作、企业工单或高可用集群为目标。

### 要解决的问题

- 服务或设备异常时，管理者需要跨多个后台和页面才能确认影响与处理入口；
- 家庭网络中的设备和服务信息分散，IP 变化或新设备出现后难以持续维护；
- 日常管理者需要的是“是否正常、哪里有问题、怎么处理”，不是大量原始指标和日志。

### 本阶段不解决的问题

- 多人权限体系、工单、企业级审计；
- 远程控制、自动修复、智能家居编排；
- 全量原始日志平台和长期监控时序库；
- 高可用或多实例部署。

默认仅服务受信任局域网。认证不是当前核心体验；如需外网访问，应由 VPN 或反向代理提供网络访问控制。未来增加有副作用的操作前，再补充轻量认证、二次确认和操作记录。

## 2. 当前能力

| 模块 | 用户价值 |
|---|---|
| 运维概览 | 汇总 PVE、Docker、ImmortalWrt 的在线状态和资源概况 |
| 节点详情 | 查看 VM、Docker 容器（含 CPU%、内存用量/上限）、路由器接口的详细运行状态 |
| 服务导航 | 分类管理常用服务管理入口 |
| 设备中心 | 通过 ARP 扫描或手工录入维护局域网资产；扫描按钮按部署能力（`arp_discovery`）显隐 |
| 健康中心 | 对设备进行 Ping、TCP、HTTP 检查，并记录离线/恢复事件 |

现有能力已经完成“看见状态”和“维护资产”。下一阶段的首要目标是把异常发现、判断与处置入口组织成闭环。

## 3. 首页改版：家庭运维指挥中心

### 3.1 目标

首页从平均展示三类资源的仪表盘，改为异常优先的指挥中心。管理者打开页面时应依次获得答案：

1. 现在是否有需要处理的问题？
2. 问题是什么、影响何处、已持续多久？
3. 应该点到哪个详情或管理入口？
4. 如果没有异常，整体基础设施是否健康、有无容量风险？

### 3.2 页面层级

```text
全局状态与最后更新时间
  └─ 需处理 / 数据源离线 / 健康检查异常 / 已纳管设备
需要处理（只在有异常时展开）
基础设施状态（PVE、Docker、ImmortalWrt）
资源风险（高 CPU、内存或重要网络接口异常） + 最近事件
快捷入口（设备、健康、节点、导航）
```

异常始终高于正常资源信息。无异常时，“需要处理”区域改为简洁的 `全部正常` 横幅，不能占据首屏。

### 3.3 首页模块要求

| 模块 | 展示内容 | 关键交互 |
|---|---|---|
| 全局状态 | `全部正常` 或 `N 项需处理`、最后更新时间、数据是否过期 | 点击摘要可进入对应区域或业务页 |
| 摘要指标 | 待处理数、离线数据源、异常检查、已纳管设备 | 前三项为状态色；设备数为中性色 |
| 需要处理 | 对象、类型、原因、持续时间、严重程度 | 跳转详情；有管理 URL 时可新标签页打开 |
| 基础设施状态 | PVE VM、Docker 容器、ImmortalWrt 的概览与最新采集时间 | 点击进入节点详情 |
| 资源风险 | 高 CPU/内存、已退出容器、接口异常 | 点击进入相应节点详情 |
| 最近事件 | 最近 5 条离线/恢复事件 | 查看全部进入健康中心 |

### 3.4 异常卡要求

每张异常卡必须包含：

- 严重程度：`严重`、`注意`、`提醒`；
- 可识别的对象和类型，例如“飞牛 NAS · HTTPS 检查”；
- 可理解的失败摘要，例如“HTTP 返回 502”“采集超时”“容器已退出”；
- `已持续 N 分钟` 或 `最后成功于…`；
- `查看详情` 主操作；设备有管理 URL 时显示 `打开管理入口` 次操作。

默认按严重程度、持续时间降序展示最多 4 项；更多异常以 `查看全部 N 项` 进入健康中心或展开列表。不得自动轮播或隐藏异常。

### 3.5 状态口径

| 状态 | 定义 | 视觉 |
|---|---|---|
| 正常 | 数据源 online 或健康检查 online | 绿色 |
| 注意 | 高资源占用、容器非 running、非关键网口 down | 橙色 |
| 严重 | 数据源 offline、健康检查 offline | 红色 |
| 未知 | 未完成首次采集/检查 | 灰色 |
| 数据过期 | 最后数据超过两个轮询周期 | 黄色 |

`unknown` 和 `offline` 必须分开：前者代表没有可靠结论，后者代表已有失败结论。数据源离线时，旧快照可作参考但必须弱化，不能显示为当前正常。

### 3.6 资源风险口径

本期可使用固定阈值：CPU ≥ 85%、内存 ≥ 85%、ImmortalWrt 可用内存 ≤ 15%。每条风险显示对象、指标、当前值和色条；无风险时保留空状态文案。Docker 容器已具备容器级资源数据（CPU%、内存占比），可在后续版本接入容器内存 ≥ 85% 的风险告警。

### 3.7 响应式与可用性

- 桌面端：摘要四列、异常两列、风险与事件两列；
- 平板：摘要两列，异常随宽度一至两列；
- 手机：单列布局，导航可收起，异常主操作始终可见；
- 支持现有深色/浅色主题；状态不只依赖颜色，还要有文字与标签；
- 所有模块必须支持加载、空、局部接口失败、未知和过期状态。

验收宽度为 390px、768px、1280px，关键内容与操作不得出现横向滚动或截断。

## 4. 告警与诊断演进

首页改版后，优先补告警闭环，而不是先增加更多数据源：

1. 外部通知：飞书、微信、Telegram 或 Webhook；
2. 告警质量：连续失败阈值、恢复通知、去重、静默窗口；
3. 可解释诊断：将认证失败、超时、DNS/TLS、权限、HTTP 状态等归类为可读摘要；
4. 处置上下文：显示最后成功时间、连续失败次数、检查目标和可跳转的管理入口。

初期不应把完整原始日志复制到 Hearth；优先展示错误分类、关键日志尾部（若安全可控）和到 PVE/Docker 等原系统的跳转链接。

## 5. 功能路线图

| 阶段 | 主题 | 交付结果 |
|---|---|---|
| P0 | 异常优先首页 | 全局摘要、异常卡、处置入口、基础设施概览、最近事件、移动端适配 |
| P1 | 告警闭环 | 外部通知、恢复通知、去重、静默与维护窗口 |
| P1 | 诊断与关联 | 设备—宿主机—VM/容器—服务入口—检查的关联，错误摘要和影响范围 |
| P2 | 历史与容量 | 状态历史、趋势、容量预警与保留策略 |
| P2 | 受控操作 | 在必要时加入重启等操作，并同时引入确认、审计和访问控制 |

## 6. 验收标准

- 异常发生时，首页首屏可看到对象、原因、状态时间和处置入口；
- 正常时，5 秒内可确认全局正常并看到关键基础设施状态；
- 管理者从首页最多两次点击可抵达异常对应的详情或管理入口；
- 单个接口失败不应导致首页白屏，已加载模块保持可用；
- 首页不新增有副作用的控制操作；
- 产品功能继续以“发现问题、定位问题、进入处置”为取舍标准。
