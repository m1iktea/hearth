# Hearth 后续计划

本文记录已确认的后续需求及当前推荐的实施方向。未完成调研或未作决定的内容明确标记为待确认，不作为当前实现承诺。

## 1. 接入米家设备

> 调研方案详见 [`docs/research/mijia-research.md`](research/mijia-research.md)。

### 需求

在 Hearth 中展示米家设备的在线状态和关键状态，并为后续告警、设备关联与可选控制预留入口。

### 推荐方案

部署 Home Assistant，并在其中安装小米官方的 Xiaomi Home 集成；Hearth 再接入 Home Assistant，而不直接实现米家云协议。

```text
米家设备 <-> 米家云 / 本地网关 <-> Home Assistant <-> Hearth
```

Home Assistant 负责小米账号 OAuth 登录、设备选择、米家云消息订阅和设备协议兼容；Hearth 将其作为一个可选的数据源，负责统一展示、告警和资产关联。

### 实施范围

1. 部署并稳定运行 Home Assistant；
2. 安装 Xiaomi Home 集成，通过小米账号 OAuth 导入需要管理的设备；
3. Hearth 新增 `homeassistant` collector，通过 Home Assistant API 获取实体和设备状态；
4. 首期仅展示设备状态、关键传感器和在线/离线信息；
5. 将选中的 Home Assistant 设备关联到 Hearth 设备中心资产。

### 暂不做

- Hearth 直接登录米家账号或直接调用米家云接口；
- 所有米家设备的控制操作；
- 为未部署 Home Assistant 的用户实现通用本地 miIO/MIoT 兼容。

### 验收结果

- 可在 Hearth 首页或设备详情看到已选米家设备的在线状态与关键属性；
- 米家设备状态变化能在合理时间内反映到 Hearth；
- 某个米家设备异常时，可关联到 Hearth 的统一告警和设备详情；
- Home Assistant 不可用时，Hearth 明确显示该数据源异常，不影响其他数据源。

## 2. 告警功能

> 接收端选型调研详见 [`docs/research/alerting-research.md`](research/alerting-research.md)。

### 需求

当健康检查、数据源或重要设备发生异常和恢复时，除站内展示外，还应主动通知家庭管理者。

告警规则本身实现难度较低；难点在于选择适合家庭场景、免费、稳定且低维护的接收端。企业场景可以使用钉钉或飞书 Webhook，但不一定适合家用。

### 基础能力范围

- 异常通知：健康检查 offline、collector offline、关键资源风险；
- 恢复通知：异常对象恢复正常后发送；
- 告警质量：连续失败阈值、去重、冷却时间和静默窗口；
- 内容：对象、原因摘要、发生/恢复时间、详情或管理入口；
- 事件记录：所有通知尝试和发送结果可追溯。

### 接收端调研方向

| 方案 | 适用性 | 需要确认的问题 |
|---|---|---|
| 飞书 / 钉钉 Webhook | 工作、团队环境成熟 | 个人家庭账户是否方便长期使用，免费额度与群机器人限制 |
| Telegram Bot | 技术用户常用，跨平台推送及时 | 网络可达性、账号与隐私接受度 |
| ntfy 自建或公共服务 | 轻量、移动端体验好、可自托管 | 公共服务的可用性；自建后的维护成本和推送可靠性 |
| Gotify 自建 | 免费、本地优先、简单 | 需要额外部署服务；iOS 推送体验与客户端能力 |
| 邮件 | 通用、无需额外 App | 即时性、垃圾邮件与 SMTP 配置成本 |
| Home Assistant 通知 | 已部署 HA 时可复用多种通知渠道 | 不应让 Hearth 的基础告警强依赖 HA |

### 调研与决策原则

1. 优先免费或可自托管，不把核心通知绑定到付费 SaaS；
2. 支持移动端及时提醒和恢复通知；
3. 配置不应要求暴露 Hearth 到公网；
4. 以通用 `webhook` 通道作为底层抽象，再适配具体平台；
5. 首个正式支持的家用通道在实际试用后确定。

### 验收结果

- 同一异常不会因为轮询而重复轰炸；
- 异常与恢复通知均可到达选定接收端；
- 通知内容可直接定位到 Hearth 的详情页面或管理入口；
- 接收端失败不会影响健康检查、事件记录或首页展示。

## 3. 设备中心与导航页互通

### 需求

设备中心新增或编辑的设备，能够在导航页中作为管理入口出现；用户不必在两个页面重复维护名称、URL 和分类。

### 产品方案

设备中心是设备资料的唯一来源；导航页保留为“服务/入口的展示与组织层”。

- 设备有 `name` 和 `url` 时，可在设备编辑页选择“显示在导航页”；
- 可选择或新建导航分类，默认建议分类为 `设备管理`；
- 导航页中来自设备中心的项目显示设备标识，并链接回设备详情；
- 设备的名称、管理 URL 或启用状态变化时，同步更新对应导航项；
- 删除设备时，提示是否一并移除关联导航项；
- 仍允许在导航页创建独立链接，它们不必关联设备。

### 数据模型建议

在 `devices` 与 `nav_items` 之间建立显式关联，例如在 `nav_items` 增加可空 `device_id`，或使用单独的关联表。不能仅凭名称或 URL 进行隐式匹配。

关联规则：

- 一个设备最多关联一个默认管理入口导航项；
- 一个设备可在后续扩展为多个服务入口，但需要明确入口名称与用途；
- 关联导航项的名称和 URL 默认由设备驱动，导航页不应产生无提示的双向覆盖。

### 实施状态（2026-07-17）

已实现：

- ✅ `nav_items` 新增可空 `device_id` 字段（幂等迁移 + 部分唯一索引，一设备最多一个导航入口）；
- ✅ 导航 CRUD 校验设备存在性（设备不存在返回 422）与唯一关联（已关联返回 409）；
- ✅ `GET /api/v1/devices/{id}` 响应附带关联 `nav_item`；
- ✅ 设备详情页支持绑定/解绑导航入口；
- ✅ 导航卡片显示设备在线/离线角标（依据 `GET /api/v1/health` 的 `last_status` 推导）。

后续待实现：

- ⏳ 删除设备时清理或提示关联导航项；
- ⏳ 设备名称/URL 变更后同步更新关联导航项。

## 建议实施顺序

1. 设备中心与导航页互通：范围明确、依赖少，能立即减少重复维护；
2. 告警接收端调研与最小 webhook 抽象：先验证家用接收体验；
3. 部署 Home Assistant 并接入米家：作为可选数据源逐步引入，不阻塞 Hearth 核心监控能力。
